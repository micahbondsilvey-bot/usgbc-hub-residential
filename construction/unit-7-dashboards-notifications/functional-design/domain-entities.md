# Unit 7 — Domain Entities

Tech-agnostic domain model for the Dashboards & Notifications unit. **One new table** —
`notification` — plus DTOs for the dashboard / pipeline read-models. All persisted columns
inherit `AuditBase` from Unit 1.

Decisions reflected (all-A from `unit-7-dashboards-notifications-design-plan.md`):
- Q1=A new per-recipient `notification` table wrapping the U1 `NotificationGateway` mock.
- Q2=A six `NotificationKind` values.
- Q3=A pure `resolveRecipients(event, ctx)` function (FL-15 subject).
- Q6=A new `GET /api/v1/admin/pipeline` (Admin-only) with cursor pagination.
- Q7=A new `GET /api/v1/reviews/assigned` grouping by review status.
- Q9=A bell icon + per-recipient `readAt` column.

## Notification

The single source of truth for one delivered (mock-delivered) notification per recipient
(BR-N1..BR-N6).

- `id: UUID` (PK).
- `kind: NotificationKind` enum:
  - `INVITATION_SENT`
  - `REGISTRATION_CONFIRMED`
  - `REVIEW_SUBMITTED`
  - `REVIEW_RETURNED`
  - `PORTFOLIO_BATCH_COMPLETED`
  - `REVIEWER_ASSIGNED`
- `recipientUserId: UUID` — soft FK to `user.id`. Indexed.
- `recipientEmail: string | null` — captured at fire time for forward-compat real delivery.
  May be `null` when the recipient is identified by `userId` only.
- `subject: text` — short human-readable subject line (already supplied by callers via the
  U1 `NotificationPayload.subject`).
- `bodyMarkdown: text` — short summary in Markdown. Generated by `NotificationsService` from
  the event context at fire time.
- `context: jsonb` — event-specific shape, e.g.
  - `INVITATION_SENT` → `{ projectId, invitationId, projectRole, expiresAt }`
  - `REGISTRATION_CONFIRMED` → `{ projectId, displayProjectId, invoiceDisplayId, totalCents }`
  - `REVIEW_SUBMITTED` → `{ projectId, reviewId, reviewDisplayId, phase }`
  - `REVIEW_RETURNED` → `{ projectId, reviewId, reviewDisplayId, phase, outcome }`
  - `PORTFOLIO_BATCH_COMPLETED` → `{ anchorProjectId, phase, summary: { submittedCount,
    skippedCount, failedCount } }`
  - `REVIEWER_ASSIGNED` → `{ projectId, displayProjectId, projectName }`
- `link: text | null` — relative FE deep-link (e.g. `/projects/<id>/review`,
  `/projects/<anchorId>/portfolio`). Computed by `NotificationsService` from the event +
  context.
- `readAt: timestamptz | null` — set when the recipient marks the notification read.
- `firedAt: timestamptz` — set on insert.
- `version: int` — increments on `markRead` (LWW contract).
- inherits `AuditBase`.

Constraints / invariants:
- `recipientUserId` not null (events with no resolvable user — e.g. `INVITATION_SENT` to a
  not-yet-registered email — are NOT persisted as `notification` rows; they continue to fire
  through the U1 mock for forward-compat. The persistence layer is purely additive.)
- Indexes:
  - `notification_recipient_idx` on `(recipientUserId, readAt, firedAt DESC)` for the bell
    dropdown query.
  - `notification_kind_idx` on `(kind, firedAt DESC)` for admin debugging.
- BR-N4 idempotence: callers may pass an `eventKey` (e.g.
  `'review-submitted:<reviewId>:<userId>'`) and `NotificationsService.persist` enforces
  per-(eventKey, recipientUserId) uniqueness via in-memory dedupe within the same fire. No
  DB unique index this build (avoid migration churn); duplicates can be cleaned up by a
  future maintenance job if needed.

## Forward-compat (NOT implemented this build)

- **Real delivery channels** (email/SMS/push). The `recipientEmail` column is captured but not
  used. Future channel adapters slot into the existing `NotificationGateway` interface; the
  persistence layer is unchanged.
- **Notification preferences** (per-user per-kind opt-out). No new schema this build.
- **Per-event-key DB unique constraint**. Idempotence is enforced in-memory within each fire;
  cross-process dedupe lands when real workers come online.
- **WebSocket / SSE push**. Bell uses 30s polling; switching to push is a future swap of the
  store's polling timer for an event-stream subscription. No FE component changes required.
